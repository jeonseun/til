## 자바 클래스 멤버

`static` 키워드를 사용하면 클래스 자체에 속하는 멤버를 정의할 수 있음

### 클래스 변수

- 동일한 클래스의 인스턴스라 해도 인스턴스 변수는 각 객체가 고유한 복사본을 가짐
- 각 객체의 인스턴스 변수는 서로 다른 메모리에 할당됨
- 클래스 변수(class variables, static field)는 메모리에 단 하나만 할당됨
- 클래스 변수는 각 객체가 아닌 클래스와 연관됨
- 클래스 변수는 해당 클래스의 모든 인스턴스가 공유함
- 클래스 변수는 해당 클래스의 모든 객체가 조작할 수 있으며 객체를 생성하지 않고도 조작 가능

예를 들어 `Car` 라는 클래스의 객체가 있을 때 각 객체마다 고유한 id를 부여하려면 인스턴스 변수로 id를 선언하고 클래스 변수로 현재 id를 지정해서 객체마다 겹치지 않도록 설정할 수 있음

```java
public class Car {
    private static currentId = 0;

    private int id;
    // 객체가 생성될 때마다 고유한 id를 부여함
    // 멀티쓰레드로 인한 동기화 문제는 일단 고려하지 않음
    public Car() {
        id = ++currentId;
    }

    public int getId() {
        return id;
    }
}
```

- 클래스 변수는 클래스의 이름과 `.` 연산자로 접근할 수 있음
  `Car.currentId`
- 클래스 변수도 객체 이름으로 접근할수는 있지만 인스턴스 변수와 혼동될 수 있으므로 권장하지 않음

### 클래스 메소드

- 메소드에 `static` 키워드를 붙이면 클래스 메소드(static method)가 됨
- 클래스 메소드는 클래스 이름과 `.` 연산자, 메소드 호출문으로 사용할 수 있음 `ClassName.method()`
  - 객체 이름으로도 호출할 수 있지만 클래스 변수와 같은 이유로 권장되지 않음
- 클래스 메소드는 객체를 생성하지 않아도 사용할 수 있음
- 클래스 메소드의 용도중 하나는 클래스 변수에 접근하기 위한 것

```java
public class Car {
    private static int currentId = 0;
    ...
    // currentId에 접근하는 정적 메소드(클래스 메소드)
    public static int getCurrentId() {
        return currentId;
    }
}
```

### 메소드 제약사항

클래스 변수, 인스턴스 변수, 클래스 메소드, 인스턴스 변수를 함께 사용할 때는 다음 제약이 적용됨

1. 인스턴스 메소드는 인스턴스 변수와 인스턴스 메소드에 직접 접근할 수 있음
2. 인스턴스 메소드는 클래스 변수와 클래스 메소드에 직접 접근할 수 있음
3. 클래스 메소드는 클래스 변수와 다른 클래스 메소드를 직접 접근할 수 있음
4. 클래스 메소드는 인스턴스 변수와 인스턴스 메소드에 접근할 수 없음
   - 인스턴스 변수와 인스턴스 메소드는 객체가 있어야 사용할 수 있으므로 생성된 객체를 통해 참조해야 함
   - 클래스 메소드는 현재 객체를 가리키는 `this` 키워드 사용불가능

### 상수

- 상수(contants)는 `static` 키워드와 `final` 키워드를 함께 사용해서 만들 수 있음
- `final` 키워드는 필드에 붙을 때 해당 필드는 더이상 수정할 수 없음을 나타냄

PI를 나타내는 상수를 선언하면 다음과 같음

```java
static final double PI = 3.141592653589793;
```

- 상수는 한번 선언하면 다시 할당할 수 없음(즉 변경 불가)
  - 변경을 시도하는 경우 컴파일 에러
- 상수의 네이밍 컨벤션은 모든 글자를 대문자로 작성하고 여러 단어의 조합일 경우 각 단어를 _ 로 연결하는 것
- 기본 데이터 타입이나 `String` 타입이 상수로 선언되는 경우 해당 상수가 사용되는 부분을 컴파일러가 해당 값을 변경함(컴파일 타임 상수)
  - 상수 값의 변경이 필요한 경우 해당 상수를 사용하는 모든 클래스를 다시 컴파일 해야함

## 자바 필드 초기화

다음과 같이 필드 선언 자체에 필드의 초기값을 제공할 수 있음

```java
public class Car {
    private int speed = 0;

    private boolean move = false;
}
```

- 초기화 표현식을 한줄에 모두 넣을 수 있다면 이 방법은 잘 동작함
- 인스턴스 변수 초기화에 일부 복잡한 로직이 필요하거나(한줄에 처리할 수 없는) 초기화 도중 예외처리 등이 필요한 경우 생성자를 써야함
- 클래스 변수의 초기화를 위한 정적 초기화 블록도 있음

> 참고로 반드시 클래스 바디 최상단에 변수를 선언할 필요는 없으며 사용전에 선언하고 초기화만 해주면 됨(물론 대부분 위쪽에 선언함)

### 정적 초기화 블록

정적 초기화 블록(static initialization block)은 다음과 같은 형싱으로 사용

```java
static {
    // 초기화할 내용
}
```

- 정적 초기화 블록은 개수의 제한이 없음
- 정적 초기화 블록은 클래스 바디의 어떤 위치에 와도 상관없음
- 런타임에서 정적 초기화 블록은 소스코드에 나타나는 순서대로 처리됨
- 정적 초기화 블록대신 정적 `private` 메소드를 사용할 수 도 있음
  - 이 경우 정적 필드를 나중에 다시 초기화할 때 메소드를 재사용할 수 있음
- 정적 초기화 블록은 클래스가 로드될 때 호출되는데 클래스가 로드되는 것은 클래스를 직접적으로 사용할 때이며 다음의 상황을 말함
  - 클래스의 객체를 생성하는 상황
  - 클래스의 정적 멤버에 접근하는 상황
  - 클래스를 상속받는 상황

### 인스턴스 멤버 초기화

필드의 값 초기화를 위해 초기화 블록과 `final` 메소드를 생성자 대안으로 사용할 수 있음

#### 초기화 블록

초기화 블록은 다음과 같은 형식

```java
// 정적 초기화 블록과 달리 static 같은 별도의 키워드가 없음
{
    // 초기화할 내용
}
```

- 초기화 블록은 클래스 내 모든 생성자에 복사됨(생성자간 동일 코드 블록 공유 가능)
- 초기화 블록은 소스코드에 나타나는 순서대로 복사되며 기존 생성자의 코드보다 먼저 실행됨

### `final` 메소드

- `final` 메소드는 하위 클래스에서 재정의 할 수 없는 메소드임
- 해당 메소드를 통해서 필드를 초기화할 경우 하위 클래스에서 해당 메소드를 재사용할 수 있는 이점이 생김
- 굳이 `final` 메소드를 쓰는 이유는 하위 클래스에서 초기화에 사용되는 메소드를 재정의할 경우 예상치 못한 문제가 발생할 수 있음
  - 예를 들어 하위 타입에서 본인의 필드의 값을 초기화에 사용하려고 메소드를 재정의 했을 경우 상위 타입의 생성자가 호출될 때는 아직 하위 타입의 생성자가 호출되지 않은 상태이므로 예상하지 못한 결과가 나올 수 있음
  - 필드 초기화에 `final` 메소드를 사용하는 것은 강제사항은 아니지만 객체 초기화에서 예상치 못한 상황을 막기위해 권장되는 사항
