## 운영체제의 역할

기본적인 컴퓨터 시스템의 구성

- 사용자
- 하드웨어(CPU, 메모리, I/O 디바이스)
- 운영체제
- 응용 프로그램

운영체제는 이러한 자원이 적절하게 사용될 수 있는 방법을 제공하여 다른 프로그램이 의미있는 작업을 할 수 있는 환경을 제공함

- 운영체제는 하드웨어와 밀접한 프로그램으로 하드웨어 자원 관리자의 역할을 함
- 운영체제는 I/O 디바이스와 사용자 프로그램을 제어하는 제어 프로그램의 역할을 함

운영체제의 구성

- 커널: 운영체제에서 항상 실행되고 있는 핵심
- 시스템 프로그램: 운영체제와 관련이 있지만 커널의 일부일 필요는 없는 프로그램
- 응용 프로그램: 커널, 시스템 프로그램 이외의 프로그램

## 컴퓨터 시스템 구성요소

일반적인 컴퓨터 시스템은 다음 요소로 구성, 다음 요소들은 버스를 통해 연결됨

- CPU
- 공유 메모리
- 장치 컨트롤러(디스크 드라이브, 오디오 장치, 그래픽 디스플레이 등)

장치 컨트롤러는 특정 유형의 장치를 담당하는 요소

- 하나의 컨트롤러에 여러 장치가 연결될 수 있음(예: USB 허브가 연결된 USB 포트)
- 장치 컨트롤러는 일부 로컬 버퍼 저장소, 특수 목적 레지스터 집합을 유지 관리(제어하는 장치와 로컬 버퍼 저장소 간 데이터 이동)
- 장치 컨트롤러마다 장치 드라이버가 있음
  - 장치 드라이버는 장치 컨트롤러의 동작에 특화되어 있음
  - 운영체제에 장치에 대한 일관된 인터페이스를 제공
  - CPU와 장치 컨트롤러는 병렬로 실행됨(메모리 사이클을 놓고 경쟁)

### 인터럽트

일반적인 컴퓨터의 작업 처리 순서

예: 입출력 수행

1. 장치 드라이버는 장치 컨트롤러의 적절한 레지스터에 값을 적재
2. 장치 컨트롤러는 레지스터의 내용 검사후 수행할 작업 결정
3. 장치 컨트롤러는 장치에서 로컬 버퍼로 데이터 전송
4. 데이터 전송 완료 시 장치 컨트롤러는 장치 드라이버에 작업 완료를 알림
5. 장치 드라이버는 읽기 요청 여부에 따라 다른 행동을 함
   1. 읽기 요청인 경우 데이터 또는 데이터에 대한 포인터 반환 후 운영체제의 다른 부분에 제어넘김
   2. 다른 작업의 경우 쓰기 완료, 장치 사용 중과 같은 상태 정보 반환

이러한 과정에서 장치 컨트롤러가 장치 드라이버에게 작업 완료를 알리는 방법이 바로 인터럽트

하드웨어는 시스템 버스를 통해 CPU에 신호를 보내서 인터럽트를 발생시킬 수 있음

인터럽트 발생 시 CPU의 작업은 다음처럼 변화함

- 인터럽트가 발생하면 CPU는 하던일을 멈추고 고정된 위치로 실행을 옮김(고정된 위치는 일반적으로 인터럽트를 위한 서비스 루틴의 시작 주소를 가짐)
- 실행을 옮긴 뒤 인터럽스 서비스 루틴을 실행
- 실행이 완료되면 인터럽트 시 중단되었던 연산을 재개

인터럽트는 적절한 서비스 루틴으로 제어를 전달하며 이를 인터럽트 정보를 토대로한 특정 루틴의 호출로 볼 수 있음 이후 이 루틴은 고유의 핸들러를 호출함

중요한 부분은 인터럽트의 발생은 매우 빈번하므로 고성능을 요구하며 이것을 위해 인터럽트 루틴에 대한 포인터의 테이블을 사용할 수 있음

- 테이블을 사용하면 중간 루틴을 둘 필요가 없으며 간접적으로 인터럽트 루틴을 호출할 수 있음
- 포인터 테이블은 일반적으로 하위 메모리에 저장됨
- 포인터 테이블에는 여러 장치에 대한 인터럽스 서비스 루틴의 주소가 있음
- 인터럽트 발생 시 인터럽트를 발생시킨 장치의 인터럽트 서비스 루틴 주소를 제공하기 위해 주소의 배열(인터럽트 벡터)이 색인됨

인터럽트 처리에 앞서서 현재 상태를 저장해야 인터럽트 처리 후 중단된 연산을 재개할 수 있음 따라서 인터럽트 루틴에는 현재 상태를 저장하는 작업이 포함됨

### 인터럽트 구현

- CPU 하드웨어에는 인터럽트 요청 라인이라는 선이 있음
- CPU는 하나의 명령을 완료할 때마다 인터럽트 요청 라인을 확인해서 인터럽트 발생 여부를 확인함
- CPU가 컨트롤러에서 인터럽트 요청 라인에 보낸 신호를 감지하면 인터럽트 번호를 읽고 이 번호를 인터럽트 벡터의 인덱스로 사용하여 인터럽트 핸들러 루틴으로 이동함
  - 인터럽트 처리에는 작업중 변경될 상태 저장, 인터럽트 원인 파악, 필요한 처리 수행, 상태 복원, CPU를 인터럽트 전 실행 상태로 복귀 등의 작업이 포함됨

인터럽트를 통해 장치 드라이버가 작업을 처리하는 것을 순서로 정리하면 다음과 같음

1. CPU: 장치 드라이버의 입출력 개시
2. CPU: 하나의 명령을 실행할 때마다 인터럽트 발생 여부 확인
3. I/O 컨트롤러: 입출력 개시
4. I/O 컨트롤러: 작업 완료 또는 에러 시 인터럽트 발생
5. CPU: 인터럽트 받으면 인터럽트 핸들러에 제어 전달
6. CPU: 인터럽트 핸들러에서 데이터 처리후 복귀
7. CPU: 인터럽트로 인해 중단된 작업 재개

추가로 보다 정교한 인터럽트 처리를 위해 다음의 요구사항이 적용됨

- 중요한 처리중에는 인터럽트가 연기되어야 함
- 장치에 대한 적절한 인터럽트 핸들러로 효율적으로 디스패치할 방법이 필요
- 운영체제가 인터럽트간 우선순위를 파악하여 적절하게 대응할 수 있어야 함

이러한 요구사항은 CPU와 인터럽트 컨트롤러 하드웨어에 의해 제공됨

대부분이 CPU는 마스킹 가능 여부에 따라 2개의 인터럽트 요청 라인을 가짐(마스킹은 특정 인터럽트를 무시하거나 차단하는 것을 뜻함)

- nonmaskable interrupt: 복구 불가능한 메모리 오류와 같은 이벤트 등
- maskable interrup: 장치 컨트롤러가 서비스 요청 등

벡터 방식으로 인터럽트를 처리할 때는 인덱스를 통해 빠르게 인터럽트 소스를 검색하기 위함인데 인터럽트 벡터의 요소 개수보다 장치가 더 많기 때문에 인터럽트 체인을 이용함

- 인터럽트 벡터의 각 원소는 인터럽트 핸들러 리스트의 헤드임
- 인터럽트 발생 시 요청을 처리할 수 있는 핸들러를 찾을 때 까지 리스트의 핸들러가 하나씩 호출됨
- 이 구조는 인터럽트 테이블을 무한정 늘릴 수 없는 현실적 제약과 인터럽트 핸들러로 디스패치하는데 오랜 검색시간이 걸리는 비효율성의 절충안

인터럽트 기법은 우선순위 레벨을 구현하므로 모든 인터럽트를 마스킹하지 않아도 우선순위가 높은 인터럽트를 먼저 처리할 수 있음

### 저장장치 구조

CPU는 메모리의 명령만 접근할 수 있으므로 명령어 실행을 위해서는 메모리에 적재해야 함

범용 컴퓨터에서는 프로그램을 메인 메모리(random-access memory RAM)에서 가져옴

RAM은 휘발성이기 때문에 컴퓨터 전원을 켤 때 실행되는 부트스트랩 프로그램(운영체제가 적재됨)을 유지할 수 없으므로 부트스트랩 프로그램은 전기적으로 소거 가능하고 프로그래밍 할 수 있는 읽기 전용 메모리(EEPROM)또는 기타 형태의 펌웨어(쓰기 작업이 자주 발생하지 않고 비휘발성)를 씀

모든 형태의 메모리는 바이트 배열을 제공함

- 각 바이트 배열은 자신의 주소를 가짐
- 메모리와의 상호작용은 메모리 주소들에 대한 적재, 저장을 통해 이루어짐
  - 적재: 메인 메모리로부터 CPU 내부의 레지스터로 한 바이트(또는 한 워드)씩 옮기는 것
  - 저장: 레지스터의 내용을 메인 메모리로 옮김
- 명시적인 적재, 저장 외에 CPU는 프로그램 카운터에 저장된 위치부터 실행하기 위해 메인 메모리 명령을 자동으로 적재함

이상적으로는 프로그램과 데이터가 메인 메모리에 영구적으로 존재하면 좋으나 다음 이유로 인해 대부분 불가능함

- 메인 메모리에 필요한 모든 프로그램, 데이터를 저장하기엔 메모리가 작음
- 메인 메모리는 전원이 끊어지면 저장된 내용이 사라지는 휘발성 저장장치

### 폰 노이만 구조 시스템

폰 노이만 구조 시스템에서의 명령-실행 사이클은 다음과 같음

- 메모리에서 명령을 인출해 명령 레지스터에 저장
- 명령을 해석(이 때 메모리에서 피연산자를 인출하여 내부 레지스터에 저장할 수 있음)
- 피연산자에 대한 명령 실행 후 결과를 다시 메모리에 저장할 수 있음

이런 과정에서 메모리 장치는 단순하게 메모리의 주소를 주소 자체로만 인식하며 주소가 어떻게 생성되었는지 어떤것을 의미하는지는 모름 따라서 메모리 주소가 생성된 경위를 무시할 수 있음

### 보조저장장치

대부분의 컴퓨터 시스템은 메모리의 확장으로 보조저장장치를 사용함 보조저장장치는 대량의 데이터를 영구적으로 보존할 수 있어야 함

- 대표적으로 HDD, NVM등을 사용함
- 대부분의 프로그램은 메모리에 적재될 때까지 보조저장장치에 저장됨
- 보조저장장치는 메인 메모리 보다 훨씬 느림

저장장치의 속도, 용량, 휘발성에 따라 더 많은 종류가 있으며 특별히 백업용도로만 사용되는 저장장치를 3차 저장장치라고 함

저장장치의 엑세스 시간 순서대로 나열하면 다음과 같음 아래로갈수록 용량은 커지고 속도를 느려짐

- 1차 저장장치(휘발성)
  - 레지스터
  - 캐시
  - 메인 메모리
- 2차 저장장치(비휘발성)
  - 비휘발성 메모리
  - 하드 디스크 드라이브
- 3차 저장장치(비휘발성)
  - 광학 디스크
  - 자기 테이프

비휘발성 저장장치는 다음 두 유형으로 구분가능

- 기계적: HDD, 광 디스크, 홀로그램 저장 장치, 자기 테이프 등
- 전기적: 플래시 메모리, FRAM, NRAM, SSD 등

기계적 저장장치는 전기적 저장장치에 비해 상대적으로 용량이 크며 바이트당 비용이 저렴하며 전기적 저장장치는 일반적으로 기계적 저장장치보다 비싸며 용량이 적고 빠름

### 입출력 구조

운영체제의 많은 부분은 시스템 안정성, 성능, I/O 관리에 할애됨

인터럽트를 활용한 I/O는 소량의 데이터에는 적합하지만 대량 데이터 이동에 쓰일 경우 높은 오버헤드를 유발할 수 있음(속도가 느린 장치에서 한 바이트마다 인터럽트가 발생) 이런 문제를 해결하기 위해 DMA(direct memory access)를 사용

DMA는 장치에 대한 버퍼, 포인터, 입출력 카운트를 세팅한 후 장치 컨트롤러는 CPU의 개입 없이 메모리에서 자신의 버퍼로, 버퍼에서 메모리로 데이터 블록 전체를 전송함(블록 전송이 완료되면 인터럽트 발생) 블록이 전송되는 중에는 CPU는 다른 작업을 수행할 수 있음

## 컴퓨터 시스템 아키텍처

범용 프로세서의 수에 따라 분류하는 컴퓨터 시스템 아키텍처

### 단일 프로세서 시스템

단일 코어를 가지는 단일 CPU를 포함하는 단일 프로세서 시스템

코어: 명령을 실행하고 데이터 저장을 위한 레지스터를 포함하는 구성요소

단일 프로세서 시스템에서는 다른 특수한 목적을 위한 프로세서도 있음디스크, 키보드, 그래픽 컨트롤러 등의 장치별 프로세서 형태로 제공될 수도 있음

- CPU는 프로세스의 명령어와 범용 명령어 세트를 실행할 수 있음
- 특수 목적 프로세서는 제한된 명령어 세트를 실행하며 사용자 프로세스는 실행하지 않음
  - 운영체제에 의해 관리되기도 하며 운영체제는 프로세서들이 수행할 태스크에 대한 정보를 보내며 프로세서의 상태를 감시함

### 다중 프로세서 시스템

일반적으로 단일 코어의 CPU로 구성되는 프로세서가 두 개 이상인 시스템

- 프로세서간에 버스, 클록, 메모리, 주변 장치를 공유하기도 함
- 다중 프로세서 시스템은 처리량이 늘어나는 장점이 있음
- 단 프로세서의 수가 N개가 된다고 해서 N배로 성능이 향상되지는 않음
  - 여러 프로세서가 하나의 작업에 협력할 경우 올바르게 동작하도록 유지하는 오버헤드가 발생하며 오버헤드와 공유 자원 경합에 예상 되는 성능 향상이 낮아짐

다중 코어가 하나의 CPU에 상주하는 다중 코어 시스템 또한 다중 프로세서 시스템에 포함됨

- 프로세서 내 통신이 프로세서 간의 통신보다 빠르므로 다중 코어 시스템이 단일 코어의 다중 프로세서보다 효율적일 수 있음
- 다중 코어 프로세서는 단일 코어의 다중 프로세서에 비해 더 적은 전력을 소비하는 장점이 있음

프로세서 관련 용어

- CPU: 명령을 실행하는 하드웨어로 중앙 처리 장치
- 프로세서: 하나 이상의 CPU를 포함하는 물리적 칩
- 코어: CPU의 기본 계산 단위
- 다중 코어: 하나의 CPU에 여러 코어가 포함되는 것
- 다중 프로세서: 여러 프로세서를 포함하는 것

## 운영체제의 작동

### 부트스트랩 프로그램

컴퓨터를 켜서 프로그램을 실행하기 위해서는 초기 프로그램을 실행해야함 이 프로그램을 부트스틀배 프로그램이라 함 일반적으로 컴퓨터 하드웨어 내에서 펌웨어로 저장

- 부트스트랩 프로그램은 운영체제를 메모리에 적재하는 방법과 해당 시스템을 실행하는 방법을 알아야 함
  - 부트스트랩 프로그램이 운영체제 커널을 찾아 메모리에 적재함으로 써 구현
  - 커널이 적재되면 시스템, 사용자에게 서비스를 제공할 수 있음
  - 시스템 데몬(커널과 생명주기를 같이하는)의 경우 커널 외부에서 제공되는데 부팅을 위해 적재되는 시스템 프로그램에 의해 적재됨

### 멀티 프로그래밍, 멀티 태스킹

하나의 프로그램이 CPU, I/O 장치 등의 리소스를 항상 100% 쓰기는 어렵기 때문에 여러 프로그램은 동시에 실행될 수 있음

사용자는 일반적으로 한번에 여러 프로그램을 실행하는데 멀티 프로그래밍은 CPU가 항상 한 개 이상의 프로그램을 실행할 수 있도록 하여 CPU 이용률을 높이는 것을 말함

멀티 프로그래밍 시스템에서 실행 중인 프로그램을 프로세스라 함

- 운영체제는 여러 프로세스를 메모리에 유지하면서 프로세스 중 하나를 선택하여 실행함
- 일부 프로세스는 I/O 작업처럼 작업 완료를 기다려야 하는 경우가 생기며 이 때 멀티 프로그래밍 시스템에서는 기다리지 않고 다른 프로세스를 실행하도록 전환함

멀티 태스킹은 멀티 프로그래밍을 보다 확장한 개념으로 여러 프로세스를 실행하되 보다 빈번하게 프로세스를 전환함

- 동시에 여러 프로세스를 메모리에 유지하기 위한 메모리 관리 방식이 필요
  - 가상 메모리 개념을 통해 물리적 메모리와 논리적 메모리를 구분할 수 있음
- 다음에 실행할 프로세스를 선택하기 위한 CPU 스케줄링 방식이 필요

동시에 여러 프로세스를 메모리에 유지하기 위해 적절한 메모리 관리 방식이 필요하며 다음에 실행할 프로세스를 선택하는 CPU 스케쥴링

## 유저 모드, 커널 모드

운영체제는 악의적인 프로그램으로 인해 다른 프로그램이 영향을 받지 않도록 보장해야 하며 운영체제에서 실행하는 코드와 사용자가 정의해서 실행하는 코드를 구분해야 함

운영체제는 모드 비트를 통해 연산 모드를 구분함

- 모드 비트 0: 커널 모드
- 모드 비트 1: 사용자 모드

사용자의 응용 프로그램 실행시에는 사용자 모드에서 동작하며 응용 프로그램이 운영체제로부터 서비스를 요청(시스템 콜)하려면 커널 모드로 전환해서 동작해야 함

- 운영체제가 컴퓨터의 제어를 얻을 때는 항상 커널 모드임
- 운영체제는 응용 프로그램으로 제어를 넘기기 전에 모드 비트를 1로 전환해서 사용자 모드로 변경해야 함

전체 시스템에 중대한 영향을 미칠 수 있는 일부 명령을 특권 명령이라 함

커널 모드가 아닐 때 특권 명령을 실행하려 하면 트랩(인터럽트의 한 종류)가 발생하며 해당 특권 명령을 실행하지 않음

- 커널 모드로 전환하는 명령 또한 특권 명령중 하나
- I/O제어, 타이머 관리, 인터럽트 관리 등도 특권 명령

> 시스템 콜: 사용자 프로그램이 운영체제가 수행할 수 있는 작업을 요청하는 방법
